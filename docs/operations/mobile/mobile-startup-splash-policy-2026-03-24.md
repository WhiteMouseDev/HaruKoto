# Mobile Startup / Splash Policy (2026-03-24)

> HaruKoto 모바일 앱의 시작 경험에 대한 별도 정책 문서. 본 문서는 스플래시/브랜딩 애니메이션과 앱 시작 시점의 원칙만 다룬다. 나머지 구조 개선은 별도 설계 문서에서 다룬다.

## 왜 분리했는가

스플래시 문제는 아키텍처 문제와 연결되지만, 사용자 체감에서는 "브랜드 애니메이션을 몇 초 보여줄 것인가"라는 UX 의사결정으로 받아들여진다. 이 이슈를 다른 구조 리팩터링과 섞으면 논의가 흐려진다.

따라서 아래 두 가지를 분리한다.

- 스플래시/스타트업 정책
- 비스플래시 구조 개선

## 현재 상태

현재 앱은 아래 세 단계가 겹쳐 시작된다.

1. `runApp()` 이전 초기화
2. 인앱 스플래시 애니메이션
3. 스플래시 종료 후 인증/프로필 분기

이 구조에서 사용자가 느끼는 것은 "스플래시가 길다"지만, 실제 원인은 단순한 애니메이션 길이만이 아니다.

## 정책 결론

브랜드 애니메이션 자체는 유지할 수 있다. 다만 "매 실행마다 고정 3초"는 유지하지 않는다.

HaruKoto의 권장 정책은 아래와 같다.

### 기본 규칙

- 네이티브 launch screen은 정적 화면으로 유지한다.
- 인앱 branded splash는 `최소 노출 시간(minimum display time)`만 둔다.
- `고정 지연(fixed delay)`는 두지 않는다.
- 인증/프로필/초기 데이터 준비가 늦다고 해서 branded splash를 무한정 유지하지 않는다.
- 준비가 길어지면 neutral loading state 또는 skeleton으로 넘긴다.

### 권장 시간

- 일반 실행:
  - 최소 노출 `900ms ~ 1200ms`
  - 권장 기본값 `1200ms`
- 브랜딩 연출이 꼭 필요한 캠페인성 빌드:
  - 최대 `1500ms ~ 1800ms`
- 첫 설치 직후 온보딩 진입:
  - 예외적으로 `1800ms` 정도까지 허용 가능

### 금지 규칙

- 매 실행마다 `3000ms` 고정 대기 금지
- 인증 판정이 끝날 때까지 branded splash 유지 금지
- 네트워크 지연을 브랜딩 애니메이션으로 가리는 방식 금지

## 타깃 UX

사용자는 아래 흐름으로 인식해야 한다.

1. 앱이 즉시 켜진다
2. 짧고 의도된 브랜드 연출이 보인다
3. 바로 로그인/홈/온보딩으로 넘어간다

현재처럼 "브랜드 연출이 끝나야 다음 로직이 시작되는 느낌"을 주면 안 된다.

## HaruKoto 권장 운영안

### 정상 경로

- 정적 launch screen
- 인앱 splash 애니메이션 시작
- 앱 준비가 빨리 끝나면 `최소 1200ms` 시점에 바로 다음 화면으로 전환

### 느린 경로

- 앱 준비가 `1200ms` 안에 끝나지 않으면
- branded splash는 종료
- neutral bootstrap view 또는 해당 목적 화면의 skeleton으로 전환

### 첫 설치 / 온보딩 진입

- 첫 설치 직후는 브랜드 경험을 조금 더 허용할 수 있다
- 다만 이 경우도 `2초`를 넘기지 않는 편이 안전하다

## 구현 원칙

- 스플래시 화면은 "준비 로직 실행기"가 아니라 "표현 계층"이어야 한다
- 시작 준비 상태는 별도 bootstrap state가 관리한다
- 스플래시는 `ready`를 기다리는 것이 아니라 `minimum display time`만 책임진다

## 비고

이 문서는 지금 당장 스플래시를 개편하자는 작업 지시가 아니다. 스플래시 논의를 구조 개선 논의와 분리하기 위한 기준 문서다.
