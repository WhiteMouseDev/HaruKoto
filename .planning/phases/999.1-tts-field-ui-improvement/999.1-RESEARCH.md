## Summary

Phase 999.1은 `apps/admin`의 `tts-player.tsx` 컴포넌트를 Select 드롭다운 방식에서 세로 스택 리스트 방식으로 리팩토링하는 순수 UI 변경이다. API 엔드포인트, DB 스키마, 4개의 편집 페이지 props 인터페이스는 모두 변경 없이 유지된다.

**핵심 발견:** 현재 백엔드의 `TtsAudio` 모델은 `(target_type, target_id, speed)` UniqueConstraint를 가지며 `field` 컬럼이 없다 — 콘텐츠 항목당 TTS 레코드가 하나뿐이다. 따라서 기존 GET `/{content_type}/{item_id}/tts` API는 "가장 최근에 생성된 단일 TTS 오디오 URL"을 반환한다. vocabulary처럼 필드가 3개(reading, word, example_sentence)인 경우에도 API 응답은 하나뿐이다. UI가 per-field 상태를 표시하려면 **현재 오디오가 어느 필드 것인지 판별할 수 없음**을 반드시 인지하고 설계해야 한다.

**Primary recommendation:** TTS 필드 리스트를 독립 컴포넌트 `TtsFieldList`로 구현하되, 현재 API 한계(단일 audio record)를 수용하는 설계를 따른다 — vocabulary의 경우 3개 필드 중 "현재 저장된 오디오"를 하나만 표시하는 방식 또는 단일 audio URL을 모든 필드 행에 공유하는 방식 중 하나를 선택해야 한다. 클라이언트 상태 리팩토링이 핵심 작업이다.

## Architecture Patterns

### 현재 구조 분석

```
tts-player.tsx
├── useTtsPlayer(contentType, itemId) 훅
│   ├── selectedField: string (단일 선택 상태)
│   ├── useQuery(['admin-tts', contentType, itemId]) → TtsAudioResponse { audioUrl, field, provider }
│   ├── audioRef: useRef<HTMLAudioElement>
│   ├── isPlaying, confirmOpen
│   └── regenerateMutation → regenerateTts(contentType, itemId, selectedField)
└── UI: Select 드롭다운 + 단일 재생/재생성 버튼
```

### 목표 구조

```
tts-player.tsx (리팩토링)
├── useTtsPlayer(contentType, itemId) — 기존 훅 재사용 또는 확장
└── TtsFieldList — 새 리스트 UI
    ├── TtsFieldRow × N (N = 해당 contentType의 필드 수)
    │   ├── 필드명 레이블
    │   ├── 오디오 상태 아이콘 (✅ / ❌)
    │   ├── 오디오 있으면: ▶️/⏸️ 버튼 + RotateCcw 버튼
    │   └── 오디오 없으면: 「オーディオなし」텍스트 + [生成する] 버튼
    └── RegenerateConfirmDialog (재사용, field 파라미터 전달)
```

### Pattern 1: 단일 오디오 URL — 복수 필드 UI 매핑

**현재 API 한계:** `TtsAudio` 테이블은 `(target_type, target_id, speed)` UniqueConstraint → 항목당 1개 오디오 레코드. `field` 컬럼 없음.

**설계 옵션:**

**Option A — 단일 오디오를 마지막 생성 필드에만 표시 (권장)**
- API 응답의 `field` 필드값(현재는 TTS에 넘긴 실제 텍스트)은 필드 식별자가 아님 — 신뢰 불가
- vocabulary 3개 필드 중 "현재 저장된 1개 오디오"를 어느 필드 행에 매핑할지 알 수 없음
- **실용적 해결:** 오디오가 있으면 `default` 필드 행에 오디오 있음으로 표시, 나머지 필드 행은 항상 "오디오 없음" + [生成する]
- 이 방식은 기존 동작과 일치 (기존 Select도 default 필드로 초기화)

**Option B — 모든 필드 행이 동일한 단일 오디오 URL 공유**
- 오디오 있으면 모든 행에 ▶️ 표시, 없으면 모든 행에 ❌ + [生成する]
- 필드 수가 3이어도 3개 버튼 모두 같은 오디오를 재생
- D-03 결정("각 필드의 TTS 유무를 한눈에 파악")과 의미론적으로 어긋남

**플래너 판단 영역:** Option A가 기존 로직과 일치하고 혼란이 적음. 단, vocabulary는 reading이 default이므로 word와 example_sentence 필드는 항상 "오디오 없음"처럼 보임. **이는 D-01 목표(빠진 것 파악)를 오해하게 만들 수 있다.** 플래너는 사용자에게 이 한계를 명시하거나 백엔드 확장(field 컬럼 추가)을 별도 phase로 deferred 처리해야 할지 결정해야 한다.

### Pattern 2: 행별 독립 재생 상태 (D-05)

현재 `useTtsPlayer`는 `audioRef`가 하나 — 단일 재생 상태. 리스트에서는 행별 독립 재생이 필요하다.

**필요한 상태 리팩토링:**
```typescript
// 현재: 단일 상태
const [isPlaying, setIsPlaying] = useState(false);
const audioRef = useRef<HTMLAudioElement | null>(null);

// 필요: 행별 상태 (playingField = null이면 아무것도 재생 중 아님)
const [playingField, setPlayingField] = useState<string | null>(null);
const audioRefs = useRef<Record<string, HTMLAudioElement>>({});
```

단 현재 API가 field별 URL을 반환하지 않으므로, 실제로는 모든 필드가 같은 `audioUrl`을 가리킨다. 행별 `audioRef`를 따로 만들 이유가 없을 수 있다 — `playingField` 상태로 "어느 행이 재생 중인지"를 추적하되 오디오 인스턴스는 하나로 유지하는 방식이 합리적이다.

```typescript
// 권장 패턴: 단일 audioRef + playingField 상태
const [playingField, setPlayingField] = useState<string | null>(null);
const audioRef = useRef<HTMLAudioElement | null>(null);

function handlePlayPause(field: string) {
  if (!audioUrl) return;
  if (playingField === field) {
    // 같은 행 — 정지
    audioRef.current?.pause();
    setPlayingField(null);
  } else {
    // 다른 행 — 기존 정지 후 새 재생 (D-05 "기존 정지 후 새 재생" 권장)
    audioRef.current?.pause();
    if (!audioRef.current) {
      audioRef.current = new Audio(audioUrl);
      audioRef.current.addEventListener('ended', () => setPlayingField(null));
    }
    void audioRef.current.play();
    setPlayingField(field);
  }
}
```

### Pattern 3: 행별 독립 regenerateMutation (D-04)

각 필드 행의 [生成する] / [RotateCcw] 버튼은 해당 field를 인자로 `regenerateTts`를 호출한다.

**옵션 A — 각 행이 자체 `useMutation` 보유 (권장)**
```typescript
// TtsFieldRow 컴포넌트 내부
const mutation = useMutation({
  mutationFn: () => regenerateTts(contentType, itemId, field),
  onSuccess: () => { /* invalidate query, auto-play */ },
});
```

**옵션 B — 부모 훅에서 단일 mutation + pendingField 상태**
```typescript
const [pendingField, setPendingField] = useState<string | null>(null);
const mutation = useMutation({
  mutationFn: (field: string) => regenerateTts(contentType, itemId, field),
});
```

옵션 A가 각 행을 독립적으로 유지하므로 더 명확하다.

### Pattern 4: confirmOpen 상태 — 행별 독립

재생성 확인 다이얼로그(RegenerateConfirmDialog)는 한 번에 하나만 열린다. `confirmField: string | null` 상태로 어느 필드에 대한 다이얼로그인지 추적한다.

```typescript
const [confirmField, setConfirmField] = useState<string | null>(null);
// confirmField !== null → 다이얼로그 open
```

### Pattern 5: i18n 키 추가 패턴

현재 `tts.fields.*` 키는 이미 모든 필드를 커버한다:
```json
"fields": {
  "reading": "読み方",
  "word": "単語",
  "exampleSentence": "例文",
  "pattern": "パターン",
  "sentence": "文",
  "japaneseSentence": "日本語文",
  "situation": "シチュエーション"
}
```

추가가 필요한 키 (3개 언어 모두):
- `tts.audioPresent` — 오디오 있음 상태 텍스트 (접근성용, 아이콘 보완)
- 아이콘만으로도 충분할 수 있음 → 플래너 판단 영역

### Anti-Patterns to Avoid

- **Select 컴포넌트 잔재 유지:** `<Select>`, `<SelectTrigger>`, `<SelectContent>` import를 완전히 제거 (D-02)
- **selectedField 상태 유지:** 단일 선택 상태 패턴을 리스트에 그대로 사용하지 않는다
- **API 응답 오해:** `TtsAudioResponse.field`는 "어느 필드의 오디오인지"가 아니라 "TTS에 넘긴 실제 텍스트"다 — 필드 식별자로 사용 불가
- **4개 편집 페이지 수정:** `<TtsPlayer>` props(`contentType`, `itemId`, `itemLabel`)는 변경 없음 — 페이지 파일 수정 불필요

## Common Pitfalls

### Pitfall 1: API field 값을 필드 식별자로 오해
**What goes wrong:** `TtsAudioResponse.field`를 "어느 필드의 오디오인지"로 해석하여 행 매핑에 사용
**Why it happens:** 필드명이 `field`이고 regenerateRequest에도 `field`가 있어서 혼동
**How to avoid:** `TtsAudioResponse.field`는 실제 TTS에 넘긴 텍스트(예: "読み方")다. `target_field` 컬럼이 DB에 없다 — 필드 식별자 매핑에 사용 불가
**Warning signs:** `record.field === 'reading'` 같은 비교 코드

### Pitfall 2: useTtsPlayer 훅 시그니처 변경으로 4개 편집 페이지 깨짐
**What goes wrong:** 훅 반환값이나 파라미터를 변경하여 vocabulary/grammar/quiz/conversation edit 페이지에서 컴파일 에러 발생
**Why it happens:** 훅이 TtsPlayer 내부에서만 쓰이는 것처럼 보이지만 편집 페이지들도 TtsPlayer를 통해 간접 의존
**How to avoid:** `useTtsPlayer(contentType, itemId)` 시그니처 유지. TtsPlayer props 인터페이스(`contentType`, `itemId`, `itemLabel`) 변경 없이 내부만 리팩토링
**Warning signs:** 편집 페이지 4개에서 TS 에러 발생

### Pitfall 3: Audio 인스턴스 누수
**What goes wrong:** 컴포넌트 언마운트 시 진행 중인 오디오가 정리되지 않음
**Why it happens:** 행별 Audio 인스턴스를 여러 개 생성하면서 cleanup 누락
**How to avoid:** `useEffect` cleanup에서 모든 audio 인스턴스 pause + null 처리. 현재 단일 `audioRef` cleanup 패턴을 유지한다
**Warning signs:** 컴포넌트 교체 후에도 오디오가 계속 재생됨

### Pitfall 4: TypeScript strict 모드 — optional chaining 누락
**What goes wrong:** `audioRef.current.pause()` 직접 호출 시 null 가능성 TS 에러
**Why it happens:** strict + noUncheckedIndexedAccess
**How to avoid:** `audioRef.current?.pause()` 패턴 일관 사용

### Pitfall 5: i18n 키 3개 언어 미동기
**What goes wrong:** ja.json에만 키 추가하고 en.json, ko.json 누락 → 컴파일 에러 (Phase 1 결정: ja.json이 IntlMessages TypeScript 타입 소스)
**Why it happens:** ja.json만 편집하는 실수
**How to avoid:** ja.json, en.json, ko.json 동시에 동일한 키 구조로 업데이트

  <Button variant="outline" size="icon" onClick={() => handlePlayPause(field.value)}>
    {playingField === field.value ? <Pause /> : <Play />}
  </Button>
  <Button variant="ghost" size="icon" onClick={() => setConfirmField(field.value)}>
    <RotateCcw />
  </Button>
</div>

<div className="flex items-center gap-2 py-2 border-b last:border-b-0">

## Open Questions

1. **vocabulary의 per-field 오디오 표시 방식**
   - What we know: DB에 field 컬럼 없음 → vocabulary reading/word/example_sentence 중 "어느 필드의 오디오"인지 API에서 구별 불가
   - What's unclear: D-03 "빠진 것만 바로 생성"을 위해 per-field 상태가 필요한데, 현재 인프라로는 지원 불가
   - Recommendation: 플래너가 사용자에게 이 한계를 명시적으로 확인해야 한다. 가장 단순한 해석은 "콘텐츠 항목 전체에 오디오가 있으면 default 필드만 ✅, 나머지는 항상 ❌ + [生成する]"로 표시하는 것. 이렇게 하면 현재 API로 구현 가능하고, 리뷰어가 항목당 여러 번 생성 클릭하면 마지막 생성 필드로 덮어써짐을 이해해야 한다.

2. **useTtsPlayer 훅 분리 vs 확장**
   - What we know: 현재 훅은 단일 field 개념으로 설계됨
   - What's unclear: 새 컴포넌트를 위해 훅을 교체해야 하는지, 확장해야 하는지
   - Recommendation: 훅의 내부 상태(selectedField → confirmField + playingField)를 변경하되, TtsPlayer 외부 공개 인터페이스(`contentType`, `itemId` 인자)는 유지. 기존 4개 편집 페이지는 `<TtsPlayer>` props를 통해서만 사용하므로 깨지지 않음.
