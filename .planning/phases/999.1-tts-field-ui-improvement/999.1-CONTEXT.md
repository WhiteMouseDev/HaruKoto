# Phase 999.1: TTS 필드 UI 개선 - Context

**Gathered:** 2026-03-27
**Status:** Ready for planning

<domain>
## Phase Boundary

TTS 플레이어의 필드 선택 UI를 드롭다운(`<Select>`)에서 세로 스택 리스트로 변경. 각 필드의 TTS 오디오 유무를 한눈에 파악하고, 빠진 필드만 바로 생성할 수 있도록 개선. 기존 TTS 재생/생성/재생성 기능은 그대로 유지하되 UI 형태만 변경.

</domain>

<decisions>
## Implementation Decisions

### 필드 리스트 레이아웃
- **D-01:** 세로 스택 리스트 — 각 필드가 한 줄씩 표시. 필드명 + TTS 상태 + 액션 버튼이 한 행에 배치. 콘텐츠 타입별 필드 수(1~3개)에 따라 높이 자동 조절
- **D-02:** 현재 `<Select>` 드롭다운 완전 제거 — 리스트가 모든 필드를 동시에 보여주므로 선택 UI 불필요

### TTS 상태 표시
- **D-03:** 아이콘 + 텍스트 방식 — 오디오 있으면 ✅ + 재생/재생성 버튼 표시, 없으면 ❌ + 「オーディオなし」 + [生成する] 버튼. Phase 4 패턴과 동일한 접근

### 생성 방식
- **D-04:** 개별 생성만 — 각 필드의 [生成する] 버튼을 하나씩 클릭. 필드 수가 1~3개라 일괄 생성의 이득이 없음. Phase 4의 단일 생성 패턴 유지

### 재생 동작
- **D-05:** 인라인 재생 — 각 행에서 ▶️ 버튼 누르면 해당 행에서 바로 재생. 별도 플레이어 영역 없이 행별로 독립 재생. Phase 4의 미니 플레이어를 행별로 분리하는 심플한 방식

### Claude's Discretion
- 행 내부 레이아웃 세부 디자인 (버튼 간격, 아이콘 크기 등)
- 재생 중 다른 행의 재생 버튼 클릭 시 동작 (기존 정지 후 새 재생 권장)
- 재생성 확인 다이얼로그 재사용 방식 (Phase 4 RegenerateConfirmDialog 재활용)
- 로딩/에러 상태 UI (Phase 4 패턴 따라감)

</decisions>

## Existing Code Insights

### Reusable Assets
- `tts-player.tsx` — 현재 전체를 리팩토링하되 재생/재생성 로직(useTtsPlayer 훅)은 그대로 재사용
- `regenerate-confirm-dialog.tsx` — 재생성 확인 다이얼로그 그대로 재사용
- `tts-fields.ts` — 콘텐츠 타입별 필드 정의. API 호출 시 field 파라미터로 사용
- `useTtsPlayer` 훅 — selectedField 상태, audioUrl 조회, regenerateMutation 등 핵심 로직 보유

### Established Patterns
- Phase 4: TtsPlayer는 contentType + itemId + itemLabel props로 편집 페이지에 삽입
- Phase 4: useTtsPlayer 훅이 TanStack Query로 오디오 URL 조회 + 재생성 mutation 관리
- 현재 단일 selectedField 상태 → 리스트 방식에서는 필드별 독립 상태 필요

### Integration Points
- 4개 편집 페이지의 `<TtsPlayer>` 컴포넌트는 props 변경 없이 내부만 리팩토링
- API 엔드포인트 변경 없음 — GET /{content_type}/{item_id}/tts와 POST /tts/regenerate 그대로 사용
- i18n에 필드별 상태 텍스트 키 추가 필요

</code_context>
